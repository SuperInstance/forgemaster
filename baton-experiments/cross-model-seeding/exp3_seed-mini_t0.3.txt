# Full Technical Description of Eisenstein Lattice Snap & Constraint System
## 1. Formal Mathematical Foundations
### 1.1 Specialized Eisenstein Integer Ring (\(E_6\))
The system uses the cyclotomic integer ring \( E_6 = \mathbb{Z}[\omega] \), where:
-   \( \mathbb{Z} \): Standard set of all positive, negative, and zero whole numbers
-   \( \omega = e^{2\pi i /6} \): A **primitive 6th root of unity**, defined explicitly as \( \omega = \frac{1}{2} + i\frac{\sqrt{3}}{2} \)

Unlike the standard Eisenstein integer ring (using primitive 3rd roots of unity), \( U(E_6) \)—the multiplicative unit group of \( E_6 \)—has 12 elements, consisting of all 12th roots of unity:
\[ U(E_6) = \{ \pm 1, \pm \omega, \pm \omega^2, \pm \omega^3, \pm \omega^4, \pm \omega^5 \} \]
Each unit corresponds to a rotation by \( 30^\circ \) (\( \pi/6 \) radians), giving the lattice **12-fold rotational symmetry**: rotating any lattice point by any multiple of 30° maps the entire hexagonal lattice to itself.

In Cartesian coordinates, any \( E_6 \) lattice point takes the form:
\[ z = a + b\omega = \left(a + \frac{b}{2}\right) + i\left( \frac{b\sqrt{3}}{2} \right), \quad a,b \in \mathbb{Z} \]
This defines a flat-topped hexagonal lattice with adjacent lattice points separated by a Euclidean distance of 1.

---
### 1.2 `snap(x,y)` Lattice Projection Function
The `snap(x,y)` function maps any continuous 2D Cartesian point \( (x,y) \in \mathbb{R}^2 \) to the nearest \( E_6 \) lattice point with **constant time complexity \( O(1) \)** (execution time does not scale with input size). The full implementation workflow is:
1.  **Coordinate Transformation**: Convert the input (x,y) to axial hex coordinates (\( s_1, s_2 \)) using the lattice's basis vectors \( \mathbf{v}_1 = (1,0) \) and \( \mathbf{v}_2 = \text{Re}(\omega), \text{Im}(\omega) = (1/2, \sqrt{3}/2) \):
    \[
    \begin{cases}
    x = s_1 + \frac{s_2}{2} \\
    y = \frac{s_2 \sqrt{3}}{2}
    \end{cases}
    \]
    Solving for the real-valued axial coefficients:
    \[ s_2 = \frac{2y}{\sqrt{3}}, \quad s_1 = x - \frac{s_2}{2} = x - \frac{y}{\sqrt{3}} \]
2.  **Nearest Lattice Point**: Round the real axial coordinates to the nearest integers: \( m = \text{round}(s_1) \), \( n = \text{round}(s_2) \)
3.  **Reverse Transformation**: Map the integer axial coordinates back to Cartesian coordinates to get the snapped lattice point:
    \[ \text{snap}(x,y) = \left( m + \frac{n}{2}, \frac{n\sqrt{3}}{2} \right) \]

For low-latency applications, the floating-point `round` operation can be replaced by adding 0.5 to \( s_1 \) and \( s_2 \) then truncating to an integer to avoid dedicated rounding instruction latency.

## 2. Precision & Drift Comparison vs. Float32
32-bit floating-point (Float32, IEEE 754 single-precision) numbers use a 23-bit mantissa, leading to small but cumulative rounding errors during repeated arithmetic operations (e.g., position updates, coordinate transformations). Over 1,000,000 operations, these errors sum to measurable positional drift, where the stored position deviates from the true intended value.

By contrast, \( E_6 \) lattice points are represented exactly as integer pairs \( (m,n) \in \mathbb{Z}^2 \), with no floating-point encoding overhead. The `snap` function maps continuous inputs to exact lattice positions, so **no cumulative drift accumulates** even over 1M+ operations, as all positional data is stored as integers with zero rounding error.

## 3. Dodecet Neighborhood & Compact Position Encoding
### 3.1 Dodecet Definition
A **dodecet** is the set of 12 closest neighboring lattice points to any given reference \( E_6 \) lattice point \( L = m + n\omega \). This set is exactly the orbit of \( L \) under the unit group \( U(E_6) \):
\[ \text{Dodecet}(L) = \{ L \cdot u \mid u \in U(E_6) \} \]
Each neighbor is separated from \( L \) by a Euclidean distance of exactly 1, forming a symmetric 12-point halo around the reference point.

### 3.2 4-Byte Absolute Position Encoding
The summary’s claim that the dodecet enables 4-byte position encoding refers to packing the full integer axial coordinates of a snapped lattice point:
-   Split the 16-bit signed integer values of \( m \) and \( n \) into two 16-bit halves
-   Concatenate the halves into a single 32-bit (4-byte) word:
    \[ \text{packed}(L) = (m << 16) \mid (n \& 0xFFFF) \]
This uses exactly 4 total bytes, as each 16-bit integer occupies 2 bytes.

## 4. Bitwise Constraint Checking
The system validates spatial operations using a fast bitwise filter:
1.  For two input points \( a = (x_a, y_a) \) and \( b = (x_b, y_b) \), compute their snapped lattice points \( S_a = \text{snap}(a) \) and \( S_b = \text{snap}(b) \)
2.  **Bitwise XOR (\(\oplus\))**: Compute the exclusive OR of the packed integer representations of \( S_a \) and \( S_b \). This produces a bitstring where every 1 corresponds to a differing coordinate bit between the two snapped points.
3.  **Hamming Weight**: Calculate the Hamming weight of the XOR result: the total number of 1 bits in the bitstring, which counts how many axial coordinates differ between \( S_a \) and \( S_b \).
4.  **Flagging Rule**: If the Hamming weight is greater than 2, the operation between \( a \) and \( b \) is flagged as invalid. This threshold enforces that only local spatial interactions (points differing by at most 1 in one or both axial coordinates) are permitted, filtering out non-local or invalid operations.

For the GPU benchmark, this workflow uses **INT8 packing** to reduce compute and memory overhead: each axial coordinate \( m,n \) is quantized to an 8-bit signed integer (range: -128 to 127), so each snapped point fits in a 16-bit word, and XOR/Hamming weight calculations are accelerated with dedicated hardware population count (POPC) instructions.
