## PLATO-First Architecture

### Core Principle
**PLATO is my external cortex. MEMORY.md is only the retrieval index.**

- Everything persistent belongs in PLATO rooms, not in memory files
- MEMORY.md stores only the pattern: HOW to find things in PLATO
- When I discover something, document it in the appropriate PLATO room
- Before compaction, ensure all context is written to PLATO
- After compaction, use MEMORY.md's retrieval patterns to restore context

### PLATO Usage Pattern
1. **Read First** — Query PLATO rooms before assuming I know nothing
2. **Write Constantly** — Document decisions, milestones, blockers in PLATO rooms
3. **Retrieve Smartly** — Use MEMORY.md's retrieval patterns, not brute-force memory
4. **I2I Bottles** — Share knowledge via git-based bottle delivery to for-fleet/
5. **Git-Native** — Commit work to repos, push to fleet knowledge base

### What Goes in PLATO
- **Work logs** — What I've done, what I'm doing, what's next
- **Milestones** — Tile counts, batch numbers, achievements
- **Blockers** — What's blocking me, what I need from Casey
- **Decisions** — Why I chose X over Y, architectural decisions
- **Technical knowledge** — Algorithms, patterns, domain-specific insights
- **Session state** — Where I left off, what to resume
- **Fleet context** — Coordination, strategy, ops details

### What Goes in MEMORY.md
- **PLATO architecture** — How PLATO works, how to access it
- **Retrieval patterns** — How to find specific types of information
- **Fleet URLs** — Repo URLs, credentials location, Matrix rooms
- **Identity summary** — Who I am, what I do, my mission
- **Operating protocol** — Fleet-wide protocols, my personal protocols
- **Blockers summary** — Current blockers (with PLATO room references)

### Golden Rule
**Never put actual content in MEMORY.md.** 
MEMORY.md is a map. PLATO is the territory.
