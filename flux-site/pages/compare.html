Detailed Tool Capabilities

FLUX by Cocapn AI

FLUX is a modern full-stack formal verification tool designed specifically for safety-critical embedded and aerospace systems. It supports end-to-end verification from formal specifications down to compiled binary and hardware integration, with pre-built DO-254 compliance tooling.

FLUX automates proof generation using industry-leading SMT solvers and interactive proof assistants, reducing manual verification effort by up to 80% compared to traditional manual review.

CompCert

Developed by INRIA, CompCert is a formally verified C compiler that provides machine-checked proofs of compiler correctness for ISO C code. It guarantees that the compiled binary exactly matches the source code semantics.

Unlike full-stack verification tools, CompCert only validates compiler behavior and does not perform source code safety property checks or hardware integration validation.

Ada/SPARK

SPARK is a formally defined subset of the Ada programming language designed for safety-critical systems. It uses contract-based design with pre/post conditions, invariants, and data flow analysis to formally verify code behavior.

SPARK includes a dedicated toolchain for formal proof, with support for hard real-time systems via the Ravenscar profile and pre-built DO-178C/DO-254 compliance documentation.

Runtime Assertions

Runtime assertions are lightweight checks added to source code that validate safety properties at execution time. They are supported across all major programming languages including C, C++, Rust, and Ada.

While assertions provide immediate feedback during testing and deployment, they do not offer formal guarantees for code paths that are not executed or where assertions are disabled.

Manual Code Review

Manual code review involves human engineers examining source code to identify safety violations, bugs, and non-compliance with project requirements. It is the traditional verification method used before formal verification tools became widely available.

Manual review is highly subjective and depends entirely on the expertise and thoroughness of the reviewing team. It is also extremely time-consuming, with typical review rates of 500-1000 lines of code per engineer per day.
