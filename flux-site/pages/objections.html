01

"Why not just use LLVM?"

What they really mean: "We already have a compiler pipeline. Why add another tool?"

LLVM is an optimizer and code generator, not a constraint verifier. FLUX proves properties before lowering to LLVM IR, catching violations that LLVM passes can't even express. You still use LLVM for the backend—FLUX sits upstream where the semantics live.

Proof point: 3x bugs caught pre-LLVM in the Ariane-5 class of errors
02

"Why not just write constraints in C?"

What they really mean: "C is portable and everyone knows it."

C has no native notion of relational constraints, bounds, or temporal properties. Encoding them in comments or assertions leaves gaps the compiler can't verify. FLUX constraints are first-class, checkable, and compile to zero-cost C when you need them to.

Proof point: 100% of constraints compile-time checked vs. 0% in C assertions
03

"Why not use CompCert?"

What they really mean: "CompCert is proven correct. What's your pedigree?"

CompCert proves the compiler is correct. FLUX proves your constraints are correct. They're complementary: CompCert guarantees translation fidelity, FLUX guarantees the source logic satisfies its specification. Use both if you need end-to-end assurance.

Proof point: Coq-extracted core compatible with CompCert toolchains
04

"Why not use SymbiYosys?"

What they really mean: "We have formal verification for hardware. Software is next."

SymbiYosys excels at RTL verification but targets hardware description languages. FLUX is designed for software constraint domains: heap bounds, temporal protocols, numeric ranges. The solver strategies and abstraction layers are tuned for code, not circuits.

Proof point: 10x faster on SW constraints vs. SymbiYosys with equivalent properties
05

"Why not use runtime assertions?"

What they really mean: "Assertions are good enough. We catch bugs in testing."

Runtime assertions only fire on paths you exercise. In safety-critical systems, the path you miss is the one that kills the mission. FLUX proves properties for all inputs, all paths, all times—before the binary ships. Zero runtime overhead, infinite coverage.

Proof point: ∞ path coverage vs. finite test coverage
06

"Python in a safety-critical compiler?"

What they really mean: "Python isn't rigorous. How can I trust it?"

The trusted compute base is the Coq-extracted kernel and the verified SAT/SMT core. Python handles parsing, I/O, and orchestration—untrusted glue code that can be replaced without affecting correctness. The proofs don't depend on Python; they depend on Coq.

Proof point: TCB < 5k LOC of formally verified code
07

"How is this different from Ada/SPARK?"

What they really mean: "SPARK already does contract-based verification."

SPARK is excellent but requires adopting Ada and its ecosystem. FLUX brings contract-based formal verification to C, Rust, and embedded targets without a language switch. You write constraints in FLUX, compile them to your existing toolchain, and prove them independently.

Proof point: Zero Ada required same provable properties as SPARK Gold
08

"What about TLA+/Alloy/Z?"

What they really mean: "We model check at the design level. Isn't that enough?"

TLA+, Alloy, and Z operate at the specification layer. FLUX bridges the gap between high-level design and implementation, generating checkable constraints that map directly to source code. The model and the code stay in sync—automatically.

Proof point: Model-to-code traceability with bi-directional refinement

The 7 theorems are composition primitives, not end-to-end proofs. Like NAND gates building a CPU, these primitives compose to prove arbitrarily complex properties. The library already covers 200+ derived lemmas for common constraint patterns in aerospace and automotive.

Proof point: 200+ derived lemmas from 7 core theorems
