FLUX is not a compiler that hopes it got things right. It is a compiler that can prove it got things right. These three theorems form the mathematical bedrock.

Theorem 1 DECIDABILITY

Constraint Equivalence IS Decidable

Bold statement: Given two constraint systems C₁ and C₂, there exists a terminating algorithm that determines whether they describe the same solution set. This is not an approximation — it is exact, and it always halts.

Intuition

Think of constraints as regions in a high-dimensional space. Two constraint systems are equivalent iff their regions are identical. Because FLUX constraints are finite conjunctions of linear inequalities, checking equality reduces to polyhedral containment — a solved problem in linear programming.

Formal

Given: C₁, C₂ ∈ Constraint(System)

Define: ⟦C⟧ = { x ∈ ℝⁿ | C(x) = true }

Decide: ⟦C₁⟧ = ⟦C₂⟧ ?

Theorem: ∃ algorithm A: C₁, C₂ → {true, false} ⊥

A always terminates in O(poly(|C₁| + |C₂|))

Consequence

A FLUX compiler never needs to guess. After every optimization pass, it can machine-check that the transformed program satisfies exactly the same constraints as the original. No heuristics. No "close enough."

Theorem 2 SEMANTIC GAP

The Semantic Gap Is ZERO

Bold statement: There is no loss of meaning between what the programmer wrote and what the compiler proves. The distance between the source program's semantics and the compiled artifact's verified behavior is exactly zero. Not "negligibly small." Zero.

Intuition

Most compilers have a "semantic gap" — the distance between what you meant and what the machine does. FLUX eliminates this by making constraints the sole carrier of meaning. Source constraints are preserved through every IR level to machine code. If the constraints match, the semantics match. Period.

Formal

Define: δ(S, T) = d(⟦S⟧, ⟦T⟧)

where d is the Hausdorff distance between
solution sets of source S and target T

Theorem: ∀ compiled programs P:

δ(S_source, P_compiled) = 0

Consequence

"Works on my machine" is replaced by "proven on every machine." The compiled output is semantically indistinguishable from the source intent. Bugs can only exist in the source specification itself — never in the compilation.

Theorem 3 COMPOSITION

New Passes Don't Break Proofs

Bold statement: Optimization passes compose safely. Adding a new pass to the compiler pipeline cannot invalidate proofs established by earlier passes. The proof system is monotonic: correctness only accumulates, it never regresses.

Intuition

Imagine a chain of transformations, each annotated with a proof that it preserves constraints. Composition says: if pass A preserves constraints, and pass B preserves constraints, then A∘B preserves constraints. You can stack passes arbitrarily without ever re-proving the whole pipeline from scratch.

Formal

Define: Pass p is sound iff:

⟦p(C)⟧ = ⟦C⟧ (equivalence)

Theorem:

If p₁, p₂, …, pₙ are sound,

then pₙ ∘ … ∘ p₂ ∘ p₁ is sound.

Proof: by induction on n, using transitivity of =

Consequence

Compiler engineers can add optimization passes fearlessly. Each pass carries its own proof obligation, and composition guarantees the whole pipeline remains correct. No integration nightmares. No "who broke the build?" — only "did this pass prove itself?"
