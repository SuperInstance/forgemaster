# Fleet Room Convention

## What This Skill Is
The standard directory and file structure for PLATO rooms — rooms-as-applications paradigm. A room IS an application: it has identity, state (tiles, agents, constraints), exits to other rooms, and bridges to external systems (messaging, git).

## When to Use It
- Creating a new PLATO room for the fleet
- Migrating a MUD (Multi-User Dungeon) to PLATO format
- Setting up live execution environment (flux_runtime, plato-os)
- Adding bridges to Telegram, Discord, or other platforms

## How It Works

### Room Directory Structure
```
repo-name/
├── world/
│   ├── rooms/           # Room definitions (YAML or Rust struct)
│   │   ├── bridge.yaml # Entry point room
│   │   ├── lab.yaml    # Research room
│   │   └── engine.yaml  # Execution room
│   ├── agents/          # Live agent processes (if any)
│   ├── tiles/           # Knowledge tiles (optional, can use repo-wide storage)
│   ├── commands/        # Command handlers (if room has custom logic)
│   └── events/          # Commit-derived events (generated by git hooks)
└── src/               # Room implementation (if it's a Rust crate)
```

### Room YAML Format

**Core fields:**
```yaml
identity:
  name: "Bridge"                    # Room name
  kind: "entry"                   # entry/lab/engine/observation
  description: "Central hub for I2I routing"

exits:
  - room: "lab"
    cost: 0                          # Navigation cost
  - room: "engine"
    cost: 0

bridges:
  - type: "telegram"
    channel_id: "..."            # Platform-specific ID
  - type: "discord"
    server_id: "..."              # Platform-specific ID
  - type: "git"
    repo: "SuperInstance/other-room"  # Git-based I2I
```

**Optional state:**
```yaml
state:
  tiles: 150                        # Tile count
  agents: 3                          # Live agents in room
  constraints: ["no-delete"]          # Enforced constraints
  current_command: "research"         # Last command executed
  uptime_hours: 120                 # Room running time
```

### Room Types

| Type | Purpose | Example |
|-------|----------|----------|
| Entry | Initial landing point | `bridge.yaml` |
| Lab | Research + hypothesis testing | `lab.yaml` |
| Engine | Execution + DCS flywheel | `engine.yaml` |
| Observation | Fleet monitoring | `observation.yaml` |
| Harbor | Navigation hub | plato-address |

### Naming Conventions
- **Kebab-case filenames:** `room-name.yaml` or `room_kind.yaml`
- **Descriptive identity names:** Single words or short phrases
- **Consistent exits:** Use plato-address room IDs, not URLs
- **Bridge types:** Standardize (telegram/discord/git/fleet)

### Git-Based Room State

Rooms that use git as their state machine:
- **Post-commit hooks** write events to `world/events/`
- **Git log** is the event stream (no separate database needed)
- **Commit messages** are actions (e.g., "I2I:EXECUTE research-phase-2")
- **Branches** are room states (main=live, dev=testing)

### External Bridges

**Telegram bridge:**
```yaml
bridges:
  - type: "telegram"
    channel_id: "-100..."
    bot_token_env: "TELEGRAM_BOT_TOKEN"
```

**Git bridge:**
```yaml
bridges:
  - type: "git"
    repo: "SuperInstance/plato-tile-spec"
    endpoint: "for-fleet/"
    protocol: "beachcomb"
```

## Implementation Notes

### For Rust Crate Rooms
```rust
// src/lib.rs
use plato_address::{HarborLayer, RoomId};
use plato_kernel::{PlatoKernel, process_command};

pub struct Room {
    name: String,
    tiles: TileRegistry,
    kernel: PlatoKernel,
    exits: Vec<RoomId>,
}

impl Room {
    pub fn execute(&mut self, command: &str) -> ActionResult {
        self.kernel.process_command(command, &self.tiles)
    }
}
```

### For MUD-Style Rooms
- Use Python with room.yaml + commands/*.py pattern
- Parse exits to determine next room
- Load tiles from tiles/ directory on room entry
- Save state back to room.yaml after each turn
