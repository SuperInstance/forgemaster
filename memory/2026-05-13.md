### Low-Rank + Hierarchical Spline (v0.6.0)

Built three compression strategies and ran honest shootout:

**LowRankLinear**: W ≈ U@V, learned factorization
- Best for: classification, sharp boundaries, categorical tasks
- topic-classify: 80% at 7,493 params (vs 100% at 20,933 dense)

**SplineLinear**: single-scale IDW from Eisenstein lattice control points
- Best for: smooth/continuous tasks (drift, sensor, regression)
- drift-detect: 100% at 162 params (20x compression, SAME accuracy)
- topic-classify: 31% — fails on sharp boundaries

**recommend_variant()**: task-aware auto-selection
- "drift", "sensor", "continuous" → spline
- "classify", "detect", "intent", "topic" → lowrank
- Deployer now uses this for auto variant selection

### Key Insight
IDW interpolation creates smooth weight matrices by construction.
This is PERFECT for smooth tasks (drift = 100%) but WRONG for sharp tasks.
Low-rank factorization lets the model learn sharp decision boundaries directly.
**Use the right tool for the right task.**

### Collective Inference (v1.0.0)
Core loop per room per instance:
1. PREDICT: at t-minus-event, I expect X with confidence Y
2. LISTEN: sensors watch for reality
3. COMPARE: prediction vs actual → delta
4. GAP: if mismatch → GapSignal → FocusQueue
5. LEARN: focus on gaps, update understanding
6. SHARE: broadcast to peers via I2I

Focus scoring: confidence × delta = "how SURE were we × how WRONG were we"
Highest focus = "we were SURE and we were WRONG" → that's the frontier

### AgentField Implementation (NEW)

Built `agent_field.py` — the shared tensor field for within-agent room coordination. 35/35 tests passing.

**Architecture:**
- One `AgentField` holds N rooms as rows in a shared state tensor [N, 9]
- Each room is a `RoomView` (index into the shared arrays)
- Coupling matrix C[i,j] = how much room i is influenced by room j
- Side-channels (nod/smile/frown) are coupling modulation, not messages
- Tick update: compute coupling forces, mix with damping, advance clock

**9-channel semantics:**
| Ch | Name | Meaning |
|----|------|---------|
| 0 | confidence | how sure this room is |
| 1 | entropy | how distributed the prediction |
| 2 | drift | rate of state change |
| 3 | focus | attention weight |
| 4 | gap | prediction vs reality delta |
| 5 | salience | importance of room output |
| 6 | coupling | how much room affects others |
| 7 | resonance | feedback strength |
| 8 | phase | predict/observe cycle position |

**Chirality state machine:** exploring → locking → locked (mirrors dodecet-encoder's temporal controller)

### Local PLATO Runtime (21:30 AKDT)

Built a three-layer knowledge system:

**Layer 1: Hot PLATO (SQLite, 16MB)**
- Synced from remote PLATO (147.224.38.131:8847)
- 14,014 tiles across 11 rooms
- Boots in ~5ms, queries in 0.1µs (room lookup)
- 10,000× faster than remote HTTP

**Layer 2: FLUX Vector Twin (29MB JSON)**
- 64-dim character n-gram hash embeddings with IDF weighting
- 14,014 tiles indexed in 18s
- Semantic search: 30ms across full corpus
- No GPU, no external model, no API calls
- "hexagonal packing" → finds related content when substring finds nothing
- Files: .local-plato/flux_vector_twin.py, .local-plato/flux_vectors.json

**Layer 3: GitHub Twin (forgemaster repo)**
- Full export of all rooms to plato-rooms/*.json
- Prunable: unload rooms not in current AgentField
- Loadable on demand: request room → pull from GitHub → boot
- Every crew that boards refines the operational manuals
- Files: .local-plato/plato_sync.py

**Architecture:**
```
Hot PLATO (SQLite, ~5ms boot, ~0.1µs queries)
  ↕ sync
Vector Twin (JSON, ~18s index, ~30ms semantic search)
  ↕ export/import
GitHub Twin (git repo, cold storage, on-demand loading)
  ↕ delta sync
Remote PLATO (HTTP, ~100ms/room, fleet shared memory)
```

**Key insight:** The 9-channel FluxVector IS a crude embedding. The vector twin extends it to 64D with IDF-weighted features. The coupling matrix operates in the same space. Semantic search and coupling dynamics are the same operation at different scales.
