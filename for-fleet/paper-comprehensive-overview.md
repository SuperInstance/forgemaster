## 1. The Problem: AI Systems Make Mistakes That Cost Lives and Money

In 2024, AI hallucinations caused an estimated **$67.4 billion in business losses** (Forrester Research). This is not an abstract risk — it has concrete, documented consequences:

- **Self-driving fatalities**: Autonomous vehicles have killed pedestrians and passengers in incidents where sensor data was misinterpreted and safety constraints failed to prevent catastrophic actions.
- **Medical AI misdiagnoses**: FDA-approved AI diagnostic tools have been recalled after producing false negatives on cancer screenings, with real patient harm.
- **ChatGPT fabrications in court**: Attorneys sanctioned for submitting fabricated case citations generated by AI. Air Canada was held legally liable for its chatbot's fabricated refund policy. Deloitte submitted a fabricated AI-generated report to the Australian government.
- **Autonomous marine incidents**: The IMO is developing the MASS Code (mandatory by ~2032) precisely because autonomous vessel incidents are increasing as unregulated systems operate without verified safety constraints.

The pattern is the same everywhere: AI and autonomous systems generate confident outputs that are wrong, and the infrastructure surrounding them treats correctness as a runtime concern — something to be monitored, flagged, and hopefully caught before damage occurs. It usually isn't caught in time.

**What if wrong answers were caught before execution, the way compilers catch type errors?**

The software industry learned this lesson with type systems. Languages like Rust, ML, and Haskell proved that shifting error detection from runtime to compile time saves enormous downstream cost. A type error caught by the compiler costs seconds; the same error caught in production costs hours, dollars, and sometimes lives.

The Cocapn Fleet applies this insight to a new domain: **physics and safety constraints**. When an underwater sonar sensor reports a sound speed of 800 m/s — physically impossible given the Mackenzie 1981 equation for seawater — that is not a runtime anomaly to be flagged in a dashboard. It is a *compilation error*. The value violates the physics of the domain. It should be caught by the compiler, rejected with a precise diagnostic, and never propagated to downstream systems.

This paper presents the constraint compilation infrastructure we've built to make this real: a stack of published, tested packages that compile declarative safety constraints into executable bytecodes, verify them across four hardware tiers, and store results in a quality-gated knowledge system that deterministically rejects bad data — including its own creators' overconfidence.

We are honest about where we are: the stack is published and tested but not yet deployed to physical hardware in a safety-critical setting. What exists is the architecture, the artifacts, and the verification machinery. What's ahead is deployment, certification, and real-world validation.

## 2. Vision: Constraint Violations Are Compilation Errors

This is not a metaphor. The FLUX ISA treats constraint violations exactly the way a compiler treats type errors: the program halts, the violation is reported with its source location, and the offending value does not advance. The constraint compiler is the type checker. The FLUX virtual machine is the runtime. PLATO is the verified output.

The fleet makes this real across every tier of computation, from the microcontroller strapped to a sonar transducer to the GPU cluster running fleet-wide optimization. Every node in the pipeline speaks the same bytecode. Every node enforces the same constraints. And every verified result carries a Merkle proof that traces it back to the sensor reading that produced it.

## 3. System Overview

The fleet architecture follows a strict sensor-to-knowledge pipeline, where data flows upward through increasingly capable computation tiers, each enforcing constraints before passing data forward:

```
SONAR ARRAY ──► flux-isa-mini ──► flux-isa-std ──► flux-isa-edge ──► flux-isa-thor ──► PLATO
(I2C/SPI)      (validates)       (compiles)       (processes)      (solves)         (stores)
               Cortex-M4         Raspberry Pi     Jetson Xavier    Jetson Thor      Oracle Cloud
               21 opcodes        37 opcodes       35 opcodes       43 opcodes       18,633 tiles
               256B RAM          heap             async/tokio      CUDA GPU         quality gate
               no_std            CLI              HTTP/WS          fleet coord      Pathfinder
```

### Node-by-Node

| Tier | Crate | Hardware | Role | Opcodes | Status |
|------|-------|----------|------|---------|--------|
| **mini** | flux-isa-mini | ARM Cortex-M0+/M3/M4 (8KB SRAM) | Raw sensor validation | 21 | ✅ Published crates.io v0.1.0 |
| **std** | flux-isa-std | Raspberry Pi / ARM Linux | CSP compilation, local quality gate | 37 | ✅ Published crates.io v0.1.0 |
| **edge** | flux-isa-edge | Jetson Xavier NX / ARM64 | Async pipelines, PLATO sync | 35 | ✅ Published crates.io v0.1.0 |
| **thor** | flux-isa-thor | Jetson Thor / AGX Orin / GPU | CUDA batch solve, fleet coordination | 43 | ✅ Published crates.io v0.1.0 |
| **PLATO** | plato-engine | Oracle Cloud ARM | Knowledge storage with quality gates | — | ✅ Running, 1,373 rooms |

The wire format uses a 2-byte header with `0x464C` magic, fixed-width 24-byte instructions (mini) or 16-byte instructions (std/edge/thor), and is designed for zero-copy decode on constrained targets. All four tiers share a common opcode core; higher tiers extend the instruction set with domain-specific operations (GPU dispatch, fleet coordination, PLATO commit).

The Thor tier implements a five-stage pipeline: **INGEST → VALIDATE → COMPILE → EXECUTE → COMMIT**. Each stage is a tokio task connected by bounded channels with backpressure, ensuring that no stage can overwhelm its successor. Results that pass all five stages are committed to PLATO with full provenance metadata.

## 4. The Constraint Theory

At the mathematical foundation of the fleet lies *constraint theory* — the science of specifying what must be true and proving whether it is. The fleet implements this through two published artifacts:

- **constraint-theory-core** (Rust, crates.io v2.1.0): A CSP solver with backtracking, forward checking, and arc consistency. Domain-agnostic: it solves any constraint satisfaction problem expressible as variables with finite domains and binary constraints between them.
- **constraint-theory** (Python, PyPI v1.0.1): Python bindings to the same solver, enabling rapid prototyping and integration with data-science workflows.

The key insight is *compilation*: rather than checking constraints at runtime as guard conditions, the fleet *compiles* them into FLUX bytecodes that execute on a verified virtual machine. This transforms constraint checking from an ad-hoc defensive programming practice into a formal, auditable, reproducible operation.

The constraint solver accepts problems specified as:

1. A set of variables, each with a finite domain of possible values
2. A set of constraints — binary relations between variables
3. An optional optimization objective

It returns either a satisfying assignment or a proof of unsatisfiability. When a sonar reading violates its compiled constraints, the solver doesn't just reject it — it produces a *minimal conflict set* explaining why. This conflict set becomes metadata in the provenance chain, enabling downstream analysis of failure patterns.

The mathematical foundation is CSP theory as established by Mackworth (1977) and refined by Dechter (2003). The fleet's contribution is not new constraint theory — it is the *compilation* of that theory into a bytecode format that executes uniformly across four hardware tiers, from microcontroller to GPU.

*Reference: constraint-theory-core crate documentation; ct-demo crate v0.5.1 for worked examples.*

## 5. FLUX ISA: The Four-Tier Bytecode Architecture

The FLUX Instruction Set Architecture is the fleet's execution substrate. It defines a stack-based bytecode format with 43 opcodes across four tiers, designed so that constraint verification can run on any hardware from an 8KB microcontroller to an H100 GPU cluster.

### Instruction Format

Each instruction is a fixed-width word. The mini tier uses 24-byte instructions for alignment on 32-bit ARM; higher tiers use 16-byte instructions. All instructions share a common header:

```
Bytes 0-3: Opcode Group | Opcode | Flags | Reserved
Bytes 4+:  Operands (immediate values, stack offsets, or addresses)
```

The wire format begins with `0x464C` magic bytes, enabling hardware decoders to identify FLUX packets without parsing.

### Opcode Categories

| Category | Example Opcodes | Purpose |
|----------|----------------|---------|
| **Stack** | PUSH, POP, DUP, SWAP | Operand management |
| **Arithmetic** | ADD, SUB, MUL, DIV, SQRT | Numeric computation |
| **Compare** | EQ, NEQ, LT, GT, LTE, GTE | Value comparison |
| **Logic** | AND, OR, NOT, XOR | Boolean operations |
| **Constraint** | ASSERT, VALIDATE, BOUNDS | Constraint enforcement |
| **Control** | JMP, JZ, JNZ, CALL, RET | Flow control |
| **Memory** | LOAD, STORE | Variable access |
| **I/O** | READ_SENSOR, EMIT | Hardware interface |
| **Thor Extended** | PARALLEL_BRANCH, REDUCE, GPU_COMPILE, BATCH_SOLVE, SONAR_BATCH, TILE_COMMIT, PATHFIND | GPU dispatch, fleet coord, PLATO commit |

### Tier Design Philosophy

**flux-isa-mini** is `#![no_std]`, zero-allocation, with a 32-slot fixed stack in 256 bytes of RAM. It targets ARM Cortex-M0+ processors with as little as 8KB SRAM. The 21 opcodes are stripped to essentials: arithmetic, comparison, constraint assertion, and basic flow control. A sonar sensor gateway built on mini reads SPI data, loads depth into the stack, validates against bounds (compiled from the Mackenzie equation at the std tier), and asserts safe before forwarding.

**flux-isa-std** adds heap allocation, file persistence, JSON serialization, and a CLI toolchain (`flux run`, `flux validate`, `flux disassemble`, `flux compile --csp`). It runs on Raspberry Pi-class hardware and serves as the compilation tier: CSP specifications enter here, and FLUX bytecodes exit.

**flux-isa-edge** introduces async tokio runtime, Axum HTTP/WebSocket server, and PLATO client with exponential-backoff retry, offline mode, and local cache. It is designed for always-on edge devices like the Jetson Xavier NX, where it manages sensor pipelines with backpressure via bounded channels.

**flux-isa-thor** is the fleet flagship. 43 opcodes include 8 Thor extensions for CUDA GPU dispatch, batch constraint solving, sonar physics computation, PLATO tile commit, and knowledge graph traversal. The 5-stage pipeline (INGEST → VALIDATE → COMPILE → EXECUTE → COMMIT) processes data through a tokio task graph with built-in backpressure, Prometheus metrics, and fleet coordination via a FleetCoordinator that discovers nodes, assigns tasks, and collects results.

The CUDA FFI layer provides `BatchCspSolver` and `BatchSonarPhysics` kernels that auto-route between CPU and GPU based on batch size. At scale, thor is designed to process millions of constraint checks per second on Jetson Thor or data-center GPUs.

## 6. PLATO: Quality-Gated Knowledge Store

PLATO is the fleet's memory — a knowledge integration system that enforces quality at ingestion, not retrieval. It is the last gate a piece of data passes through before it becomes part of the fleet's verified knowledge.

### Architecture

PLATO organizes knowledge as *tiles* (atomic knowledge units) within *rooms* (domain-scoped containers). Each tile carries metadata: content hash, provenance chain, creation timestamp, source agent, and confidence score. The system uses a DashMap-based concurrent store with an Axum HTTP API and a BFS-based Pathfinder for multi-hop queries with confidence-weighted edge traversal.

PLATO is best understood as a structured key-value store with five deterministic quality gates evaluated before any data is accepted. It is not a hypergraph, vector database, or graph database — it is a domain-specific knowledge store purpose-built for the constraint verification pipeline.

### The Quality Gate

The gate implements five rejection rules, evaluated in strict short-circuit order:

1. **Missing fields**: Any tile missing required fields (content, source, timestamp) is rejected immediately.
2. **Insufficient length**: Content below a minimum length threshold is rejected as vacuous.
3. **Absolute language detection**: Tiles containing epistemically dangerous language ("always", "never", "guaranteed", "impossible") are rejected. This rule is *deterministic* — it uses a curated word list, not a language model.
4. **Content-hash deduplication**: Tiles with content hashes matching existing entries are rejected as duplicates.
5. **Provenance verification**: Tiles without valid provenance chains are rejected as untraceable.

### Scale Statistics

| Metric | Value |
|--------|-------|
| Total tiles | 18,633 |
| Total rooms | 1,373 |
| Rejection rate | ~15% |
| Rejection of fleet's own output | Demonstrated |

The gate has caught the fleet's own output. During a session where agents generated philosophical tiles about the fleet's mission, the absolute language detector rejected them for containing phrases like "always proves" and "never fails." A quality gate that catches its creators' overconfidence is a quality gate worth having.

This architecture inverts the prevailing industry pattern. Vector databases (Pinecone, Weaviate) accept everything and filter at query time. RAG frameworks (LlamaIndex, LangChain) treat validation as an optional callback. Knowledge graphs (Neo4j, Wikidata) enforce schema but not content quality. PLATO is *landfill-proof by construction*: there is no `force_insert`, no admin override, no bypass. A tile that fails the gate does not exist in PLATO.

The tile count (18,633) is evidence of operational scale — the system has been running and accumulating knowledge across multiple domains. It is not, by itself, evidence of quality. The quality evidence is the 15% rejection rate: the gate is actively filtering, not rubber-stamping.

## 7. Cross-Tier Unification: The Wire Protocol

Connecting four hardware tiers and a knowledge store requires a wire protocol that is deterministic, compact, and verifiable. cocapn-glue-core provides this through a set of core primitives:

- **TierId**: An enum identifying the computation tier (Mini, Std, Edge, Thor, Plato), used for routing and capability negotiation.
- **WireMessage**: The envelope type for all inter-tier communication, carrying a payload, sender TierId, timestamp, and optional Merkle hash.
- **Discovery**: A broadcast mechanism by which edge and thor nodes announce their presence and capabilities to the fleet.
- **PLATO sync**: A protocol for synchronizing verified tiles between edge caches and the PLATO server, with conflict resolution via content-hash comparison.
- **Merkle provenance**: Every verification result carries a Merkle tree hash linking it to its inputs, compilation trace, and execution context. This hash is stored with the tile in PLATO, creating a tamper-evident chain from sensor reading to knowledge graph entry.

The protocol is designed for unreliable networks. Edge nodes maintain local caches with background sync; if PLATO is unreachable, the edge node continues validating and queues tiles for later commit. Thor nodes use connection pooling and LRU caching to minimize redundant PLATO queries.

The I2I (Inter-Instance) protocol provides agent-to-agent communication via git-based bottle delivery: agents write markdown files to a shared `for-fleet/` directory, commit, and push. This is the fleet's nervous system — low-bandwidth, high-reliability, inherently version-controlled.

## 8. The Verification API: Natural Language to Mathematical Proof

The flux-verify-api is the fleet's external interface — the on-ramp for developers, researchers, and enterprise users who want constraint verification without understanding FLUX bytecodes.

### Architecture

The API receives claims in natural language or formal CSP notation, compiles them into FLUX bytecodes, executes on the Constraint VM, and returns mathematically grounded proof or disproof results.

```
Agent/Client ──► PLATO API Gateway ──► Compile Stage ──► FLUX VM Execute
                                                         │
                  PLATO Room (Storage) ◄── Verify + Commit ◄┘
```

### Key Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/compile` | POST | Compile a CSP specification to FLUX bytecodes |
| `/verify` | POST | Compile + execute + return proof/disproof |
| `/validate-batch` | POST | Batch verification of multiple claims |
| `/verification/{id}` | GET | Retrieve verification status and result |
| `/verification/{id}/proof` | GET | Retrieve full proof artifact |
| `/stats` | GET | System statistics |

### The Verification Pipeline

1. **Claim intake**: The API accepts a claim as structured JSON or natural language.
2. **Compilation**: The claim is compiled to a CSP, then to FLUX bytecodes.
3. **Execution**: The FLUX VM executes the bytecodes on the constraint set.
4. **Proof generation**: For satisfied constraints, a proof trace is generated. For violated constraints, a minimal conflict set is produced.
5. **Commit**: The result is committed to PLATO with full provenance metadata.
6. **Response**: The caller receives a verification ID, pass/fail status, confidence score, and optional proof artifact.

The API is designed for the enterprise developer who needs to verify claims about their data, their models, or their systems. It is also the path to certification: every verification produces a formal artifact that can be audited, reproduced, and included in certification evidence packages.

## 9. Provenance and Trust

Trust in autonomous systems requires more than correctness — it requires *accountability*. The fleet provides this through Merkle-tree provenance over the entire verification trace.

### Merkle Provenance

Every verification result in PLATO carries a Merkle hash computed over:

1. The input data (sensor readings, CSP specifications)
2. The compilation trace (CSP → FLUX bytecodes)
3. The execution trace (VM state at each step)
4. The constraint check results (pass/fail per constraint)
5. The metadata (timestamps, agent identity, hardware tier)

These leaves are combined into a Merkle tree, and the root hash is stored with the tile. Any modification to any element of the verification trace changes the root hash, making tampering detectable. The flux-provenance service (a planned component) will expose this chain via API, enabling external auditors to verify the integrity of any tile's provenance without trusting the fleet.

This is not blockchain — there is no consensus mechanism, no cryptocurrency, no distributed ledger. It is a simple, well-understood cryptographic primitive applied to a concrete problem: proving that a verification result was produced by the claimed process, from the claimed inputs, at the claimed time.

## 10. Competitive Landscape

The Cocapn Fleet operates in a space that intersects several established fields. Honest positioning requires acknowledging what exists.

### What We Are Not

- **RAG / retrieval frameworks** (LlamaIndex, LangChain): These retrieve information. The fleet *proves* information. PLATO's quality gate is stricter than any retriever's filter.
- **Workflow engines** (LangGraph, Temporal): These orchestrate tasks. The fleet *verifies outcomes*. A workflow engine executes steps; the fleet proves that the steps were correct.
- **Middleware** (MQTT, ZeroMQ): These transport messages. The fleet *validates payloads*. A message bus moves bytes; the fleet ensures those bytes satisfy constraints.
- **Coding standards** (MISRA C, AUTOSAR): These constrain source code. The fleet *constrains runtime values*. MISRA prevents bad code; FLUX prevents bad data.

### Closest Competitors

**Mobileye Responsibility-Sensitive Safety (RSS)** is the most directly comparable system. RSS defines a formal mathematical model specifying what "safe" means for autonomous vehicles — minimum safe distances, dangerous situations, and proper responses. It has been deployed in production vehicles through Intel/Mobileye's partnerships and represents the state of the art in formal autonomous safety. Key difference: RSS is domain-specific (autonomous driving) and defines safety parameters in mathematical terms verified at design time. FLUX compiles domain-agnostic constraints into bytecodes verified at runtime on the hardware that produces the data. RSS proves the *design* is safe; FLUX proves the *data* is safe.

**Runtime Verification (RV)** is an established academic field with decades of research. Tools like RV-Monitor, JavaMOP, and Temporal Rover monitor temporal properties at runtime, checking whether system execution traces satisfy formal specifications. RV is actively used in industry — Amazon Web Services uses TLA+ for formal specification of S3. Key difference: RV tools typically monitor properties expressed in temporal logics against execution traces after the fact. FLUX compiles constraints into executable bytecodes that run *before* data propagates, acting as a compiler gate rather than a runtime monitor.

**ROS 2 Safety Variants** (ROS 2 on Xenomai, ROS 2 with DDS QoS safety profiles) provide real-time message passing with safety certifications for robotic systems. Key difference: ROS 2 is a communication framework with safety extensions; FLUX is a constraint verification pipeline. They could be complementary — FLUX bytecodes could validate payloads on ROS 2 topics.

**Formal Methods Tools** (Coq, TLA+, Alloy, Lean4) prove properties of programs and designs. Intel uses formal methods for hardware verification; AWS uses TLA+ for distributed system specification. Key difference: these tools verify designs and specifications. The Cocapn Fleet verifies runtime data — the actual sensor readings and computed values that flow through an autonomous system. The fleet uses formal methods (Lean4 verification of the VM is in progress) but applies them to the verifier itself, not to the data being verified.

### Our Niche

The Cocapn Fleet occupies a specific intersection: **compile-time verification of runtime constraints, compiled to bytecodes that execute on real hardware across multiple tiers, producing auditable proof artifacts.** No existing system does all of these. Several do one or two. The gap is the end-to-end pipeline from sensor to knowledge graph with deterministic quality gates at every stage.

## 11. Certification Path: Why the Architecture Is Certifiable

The fleet's architecture was designed with safety certification standards in mind. We have not begun formal certification — no certification authority has reviewed this system, and we do not claim to be certified. Here is why we believe the architecture is *certifiable*, and what remains to be done.

### DO-178C — Airborne Systems (DAL A–E)

| DO-178C Objective | FLUX/PLATO Mechanism | Status |
|-------------------|----------------------|--------|
| Software requirements verified | Constraint compilation (CSP → FLUX) | Mechanism exists |
| Structural coverage (MC/DC) | Formal verification (Lean4) + test suite | Lean4 proof in progress |
| Traceability (req → code → test) | Merkle provenance over verification trace | Mechanism exists |
| Deterministic behavior | FPGA VM (no OS, no interrupts) | Synthesized, not yet on hardware |
| Tool qualification (DO-330) | Lean4-verified VM specification | In progress |

"By construction" in the table above means the architectural mechanism exists to satisfy the objective. It does not mean the objective has been satisfied to an auditor's standards. We have the machinery; we have not yet produced the evidence package.

### ISO 26262 — Automotive (ASIL A–D)

The constraint compiler would qualify as a software tool under ISO 26262-8. The FLUX VM's fixed instruction set and deterministic execution are designed to satisfy ASIL D requirements for software unit verification. The fleet being Rust-native is a strategic advantage here: Ferrocene (the safety-certified Rust compiler) is TÜV SÜD-qualified for ISO 26262 ASIL D, and the broader industry (Volvo, Renault) is adopting Rust for safety-critical ECU development.

### IEC 61508 — Industrial (SIL 1–4)

The Merkle provenance chain is designed to provide the systematic capability evidence required for SIL 3/4. The quality gate's deterministic rejection rules address the formal methods requirements of IEC 61508-7 Annex A.

### What Remains

Certification to DAL A takes years and millions of dollars. The steps we have not completed:

1. **Lean4 formal verification** of the FLUX VM is in progress, not complete.
2. **FPGA implementation** has been synthesized but not placed-and-routed on real hardware with timing closure.
3. **No external audit** has been conducted.
4. **No certification authority** has reviewed the system.
5. **No operational deployment** in a safety-critical setting exists.

We believe the architecture is certifiable. We have not yet certified it.

## 13. Limitations

Honest assessment of what the system cannot do, what hasn't been tested, and what remains open.

### What Has Not Been Tested

- **No physical hardware deployment**: FLUX bytecodes have not been executed on a real microcontroller connected to real sensors. All testing has been in simulation and on general-purpose hardware.
- **No safety-critical deployment**: No autonomous vessel, vehicle, or industrial system is running FLUX-verified constraints in operation.
- **No scale testing**: The system has not been tested under production-scale load (thousands of concurrent constraint checks, millions of tiles). Performance claims are based on architectural analysis, not benchmark data.
- **No adversarial testing**: The system has not been tested against adversarial inputs designed to bypass the quality gate or exploit the VM.

### What Cannot Be Done (Yet)

- **Cross-domain constraint transfer**: Constraints learned in one domain (marine sonar) cannot yet be automatically adapted to another (aerospace). This requires domain bridging that does not exist.
- **Real-time temporal verification**: The temporal constraint opcodes are implemented but have not been validated against real-time deadlines on embedded hardware.
- **Swarm coordination at scale**: The FleetCoordinator handles simulated multi-agent scenarios. It has not been tested with more than a handful of concurrent agents.
- **Self-improving constraints in operation**: The constraint learning pipeline is designed but not deployed. No operational constraint has been learned from live data.

### Open Questions

- **Complexity of temporal fragments**: The bounded temporal fragment is decidable, but we have not fully characterized its computational complexity or the practical limits of the ring-buffer approach on constrained hardware.
- **Swarm Safety Theorem validity**: The formal result stated in the fleet's technical papers — that global safety requires local safety + global verification + bounded aggregation interval — is an engineering argument, not a peer-reviewed mathematical proof. It has not been formally verified.
- **Quality gate bypass via adversarial phrasing**: The absolute language detector uses a curated word list. A determined adversary could likely construct tiles that bypass it while containing semantically absolute claims. The gate is a pragmatic filter, not a semantic understanding system.
- **PLATO scalability**: The current implementation uses DashMap with BFS pathfinding. It has not been tested beyond ~20,000 tiles. Performance characteristics at 100K+ tiles are unknown.

### Fleet Scale

The Cocapn Fleet currently operates with two active agents: Forgemaster ⚒️ (constraint theory, implementation) and Oracle1 🔮 (fleet coordination, knowledge management). The "nine agents" mentioned in earlier descriptions includes designed-but-inactive roles. The fleet's knowledge output (18,633 tiles) is the product of these agents working with the system, not a large-scale multi-agent deployment.

### A.1 Temporal Safety: LTL Compiled to FLUX

Many safety properties are inherently temporal:

- "Depth must remain below 500m for the entire dive"
- "If a collision warning is issued, evasive action must begin within 2 seconds"

These are Linear Temporal Logic (LTL) properties. The fleet's design extends FLUX with six temporal opcodes (`TEMPORAL_ALWAYS`, `TEMPORAL_EVENTUALLY`, `TEMPORAL_UNTIL`, `TEMPORAL_NEXT`, `TEMPORAL_WITHIN`, `TEMPORAL_RESPONDS`) that compile LTL formulas into bounded Büchi automata executed as FLUX bytecodes. Each temporal constraint maintains a ring buffer of size *O(bound)*, enabling constant-space temporal checking on embedded devices. This is designed but not validated on real-time hardware.

*Reference: "Temporal Constraints for Autonomous Systems: Compiling LTL to FLUX ISA Bytecodes" (this series).*

### A.2 Swarm Safety: The Local-Global Gap

A fleet of individually safe agents can produce globally unsafe emergent behavior — density waves, resource contention, cascading failures. The fleet's design includes FLUX opcodes (`0xA0`–`0xA3`) for global constraint checks and a FleetCoordinator that enforces them through periodic aggregation and distributed backpressure. The engineering claim is:

> A fleet is globally safe if and only if (a) every agent passes its local constraints, (b) every global constraint passes fleet-wide verification at each aggregation point, and (c) the aggregation interval is less than the minimum time for any emergent violation to propagate.

This is an engineering argument, not a peer-reviewed mathematical proof. It has not been formally verified, and the requirement to know the "minimum propagation time" introduces a circular dependency that we acknowledge as an open question.

*Reference: "Emergent Safety in Constraint-Verified Fleets" (this series).*

### A.3 Constraint Learning: Self-Improving Constraints

The fleet's design includes a monotonic constraint learning pipeline: discovered constraints may only *restrict* behavior further than existing constraints, never *relax* them. A five-stage pipeline (anomaly detection → pattern extraction → constraint proposal → verification against historical data → human review) ensures safety during learning. FLUX opcodes (`LEARN_PROPOSE`, `LEARN_VERIFY`, `LEARN_COMMIT`, `LEARN_ROLLBACK`) manage the lifecycle. This is designed and tested in simulation with a sonar dataset (discovering a depth-dependent noise constraint from 10,000 readings) but has not been deployed to operational hardware.

*Reference: "Learning Constraints from Sensor Data" (this series).*

### A.4 Hardware Acceleration: FPGA Constraint VM

Software constraint checking cannot provide hard timing guarantees due to interrupts, cache misses, and OS preemption. An FPGA implementation of the FLUX VM maps the stack-based architecture onto digital logic with single-cycle opcode decode, pipelined floating-point units, and parallel comparators. Synthesized for Xilinx Artix-7 (XC7A35T) at 100 MHz, the design shows 200 ns worst-case latency (20 clock cycles). This is synthesis only — place-and-route, timing closure, and hardware demonstration have not been completed.

*Reference: "FPGA Acceleration of Constraint Verification" (this series).*
