## Cross-Model Consensus (All Three Agree)

1. **The FPGA demo must ship.** Not a paper — actual synthesized hardware with measured numbers.
2. **Lead with physical guarantees, not speed.** Speed is commodity; determinism is the moat.
3. **Need a constraint specification language / DSL.** Hand-written FLUX bytecode is not adoptable.
4. **Avionics is the correct first domain.** DO-178C + DO-254 synergy, mature FPGA safety guidance.
5. **N-Queens is not proof-of-value.** Need real industrial problems (motor controller, flight envelope).

## Unique Insights Per Model

### Claude Opus 4 — The Strategist
- "Every layer of abstraction between a constraint and its enforcement is a place where violations can hide"
- Competitive landscape is effectively **empty** — no one combines all four pieces (ISA + knowledge + hardware + cert path)
- Need to solve compilation gap: specification → FLUX bytecode is the **product**
- Risk: "LLMs solve this" narrative — counter with DO-178C regulatory moat
- 90-day plan: FPGA demo → PLATO-FLUX connection → job-shop scheduling benchmark

### Kimi — The Provocateur
- **"You're building a faster assert(), not constraint elimination"**
- Assert-halt is a failure mode, not a guarantee
- BitmaskDomain benchmark is misleading — optimized for problem sizes that don't occur in production
- The real problem: **certification cost**, not speed
- **"Synthesis, not checking"** — compile constraints to hardware state machines with no path to violation
- The product is a **silicon compiler with a proof certificate**, not a VM
- Kill three tiers. Kill PLATO integration. Pick one execution context where physics matters.
- Key reference: seL4, CompCert, Kami/Bluespec

### Nemotron — The Certification Expert
- Detailed DO-178C DAL A path: 24-36 months, 6 phases
- Most dangerous assumption: all constraints fit in 64-bit bitmask
- Need: bounded-stack hardware, pipelined comparators with EDC, TMR-protected FSM
- Power-domain isolation, separate clock domains for safety fabric
- Formal semantics for every opcode (Isabelle/HOL or Coq)
- DSL with proof certificates that feed directly into DO-178C evidence
- DO-254 for hardware + DO-178C for software = joint safety argument
- Parameterizable mask width (64/128/256) with automatic tiling

## Synthesis: What We Should Actually Do

The three models converge on a clear action plan:

### Immediate (30 days)
1. Port FLUX mini to Arty A7 FPGA with real synthesis
2. Measure actual violation detection latency (not projected)
3. Define hardware safe-state behavior (not just halt — fallback mode)
4. Write formal semantics for all 43 opcodes (Lean4 or Coq)

### Short-term (60 days)
1. Design constraint DSL that compiles to FLUX with proof certificates
2. Implement parameterizable mask width (64/128/256 with tiling)
3. Add TMR on control FSM, EDC on mask lines, separate power domain
4. Select first industrial demo: robotic arm collision avoidance

### Medium-term (90 days)
1. Ship working demo: real hardware, real constraint, real proof
2. Target avionics anchor customer (Tier 2 supplier)
3. Begin DO-254 + DO-178C joint evidence package
4. Pivot from "constraint checking" to "correctness-by-construction"

### The Pivot Point
Kimi's insight is the sharpest: the future isn't "FLUX ISA executes constraints faster."
It's **"compile constraint specifications to hardware state machines that have no path to violation."**

The VM is a prototype. The product is a silicon compiler with a proof certificate.
