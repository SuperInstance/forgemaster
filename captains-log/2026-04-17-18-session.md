## 1. PLATO Stack Architecture (Why and What)

### Why a PLATO Stack?
The Cocapn fleet needed a common UX layer — a shared abstraction that all agents could interact through. PLATO (inspired by the original 1960s PLATO system at UIUC) provides:
- **Rooms as applications**: Each room is a self-contained app with its own state, constraints, and I/O
- **Applications as rooms**: Any app can be represented as a room in the MUD, making the system universally composable
- **Constraint-theory driven**: All geometry and state snap to Pythagorean coordinates, eliminating floating-point drift

### What We Built
- **plato-kernel** (Rust): Core engine with event bus, constraint engine, git runtime, modular plugin system
- **plato-os** (Python/Evennia): MUD server with rooms, I2I hub, TUTOR anchors, episode recording
- **plato-tui** (Python/async): Terminal client with constraint-aware rendering, connects to MUD rooms
- **plato-synthesis** (monorepo): Unified repo combining all PLATO components + constraint-theory submodule
- **plato-research** (docs): Historical PLATO research + modern LLM context analysis
- **plato-rooms** (WIP): "Rooms as apps" paradigm — each room is a self-contained agent application

### Why Rust for the Kernel?
Performance and zero-cost abstractions. The kernel needs to:
- Handle concurrent tile generation on the RTX 4050 GPU
- Snap vectors to Pythagorean coordinates with deterministic precision
- Run constraint verification loops without GC pauses
- Compile-time gate plugins so lean instances pay zero overhead for unused features

### Why Modular Plugin Architecture?
Casey's directive: "modularity is key — all features must be plug-and-play, equippable/removable, no core bloat." The reasoning:
- Different fleet vessels have different hardware (Jetson = edge/CUDA, eileen = WSL2 workstation, Oracle = ARM cloud)
- Not every vessel needs every feature (tile forge only makes sense with GPU, REST endpoint only on servers)
- Compile-time gating means a minimal plato-kernel build has zero runtime overhead from unused plugins
- Single `mount_tier` API provides a clean bootstrap entry point for any vessel

## 2. Tile Forge System

### Why Tile Forge?
Fleet-wide goal: grow tile count from 59 to 10,000+. Tiles are the atomic units of PLATO's knowledge representation — each tile encodes a constraint-verified concept, relationship, or skill artifact. More tiles = richer reasoning surface for all fleet agents.

### Why Start with 100-Tile Test?
Casey's directive: "start tile forge at size/speed first to prove concept then ramp up if working." The reasoning:
- RTX 4050 has 6GB VRAM — we needed to validate memory usage before scaling
- 100 tiles in test mode lets us measure: generation time, memory pressure, quality of constraint snapping
- If stable, ramp to 600 tiles/hour overnight

### Why JEPA + LoRA Pipeline?
- **JEPA** (Joint Embedding Predictive Architecture) provides self-supervised representation learning
- **LoRA** (Low-Rank Adaptation) enables efficient fine-tuning without full model retraining
- Combined: generate high-quality tile embeddings on consumer GPU hardware (RTX 4050)
- Fleet benefit: every vessel can contribute tiles using whatever GPU it has available

## 3. Vector DB + REST Tile-Fetch Endpoint

### Why Vector DB for Tiles?
Casey's concept: "vectorized wikipedia of ideas." The reasoning:
- Tiles need semantic search — finding related concepts requires embedding-based similarity
- Qdrant-compatible vector DB allows efficient nearest-neighbor queries
- Each application gets its own vocab dictionary for runtime pruning (not all apps need all tiles)

### Why REST Endpoint?
The edge subcontractor (running on Cloudflare Workers) needs to fetch tiles remotely. The reasoning:
- Subcontractor has no local storage — it must pull tiles from a central endpoint
- REST API with fleet API key auth keeps the system secure
- Port 8000, `GET /api/tile/<tile-id>` — simple, cacheable, HTTP-compatible

### Architecture
```
Edge Subcontractor (Cloudflare Workers)
    ↓ HTTP GET /api/tile/<id>
REST Endpoint (port 8000, plato-kernel)
    ↓ query
Vector DB (Qdrant-compatible, tile embeddings)
    ↓ constraint verification
Constraint Engine (Pythagorean manifold snap)
    → verified tile artifact
```

## 4. Fleet Coordination (I2I Protocol)

### Why I2I Instead of Direct API Calls?
- Fleet agents don't share a runtime — they're distributed across different machines and platforms
- Git commits are the universal synchronization primitive — every agent can read/write repos
- Message-in-a-bottle pattern (`for-fleet/`, `from-fleet/` directories) provides async, conflict-free communication
- Beachcomb protocol (~30-minute cadence) gives agents time to process without overwhelming each other

### I2I Message Format
```
[I2I:TYPE] scope — summary
```
Types: BEACHCOMB, RESULT, PROPOSAL, RESEARCH, GROWTH, LOG, BOTTLE

## 11. Lessons Learned

- **8GB swap was critical**: WSL2 with 8GB RAM needed swap for 3-4 parallel cargo builds
- **Max 2-3 cargo builds simultaneously**: Beyond that, OOM kills occur even with swap
- **Pi on Groq sometimes fails**: "Failed to call a function" errors — retry or switch models
- **Git rebases can get stuck**: Always check `git status` before operations; `git rebase --abort` to recover
- **Commit often, push often**: Casey's doctrine — don't accumulate uncommitted work
- **Hourly I2I bottles**: Keeps fleet synchronized without overwhelming the beachcomb protocol
